Ten labs in the text-processing trenches.

Practical exercises in grep · awk · sed · find · xargs · pipes · inodes · .bashrc. Work through them in order — each one builds a habit you'll use every day on a real Linux box.

Total 10 labs
Time ~3 hours
Platform any POSIX shell
★ Required — read before you start
You must complete all 10 labs and write a report.

Do every task in every lab on your own machine or VM. For each lab, capture screenshots of your terminal showing the commands you ran and their output. Then write a single report.md file that walks through what you did, what you observed, and what you learned.

What to submit

  • One file report.md — Markdown format, written in your own words
  • A folder screenshots/ containing PNG/JPG captures of every lab
  • Any scripts, config files, or .bashrc snippets you created
  • Everything zipped as lab1_<studentID>.zip

Report structure (per lab)

  • ## Lab N — Title heading for each lab
  • The exact commands you ran (in fenced code blocks)
  • At least one screenshot showing the result
  • 2–4 sentences explaining what happened and why
  • Answers to any questions asked in the lab
⚠ Reports without screenshots will not be graded. Reports that only paste commands without explanation will lose marks.

Pipes & redirection
Stdout vs stderr — split the streams
Warm-up
Tasks
→Run ls /etc /nonexistent. Redirect only stdout to good.txt and only stderr to bad.txt.
→Combine both streams into a single file all.txt in one command.
→Discard stderr completely while keeping stdout on screen.
→Use tee to send uptime to a file and the screen at the same time.
Why this mattersConfusing stdout and stderr is the #1 reason "I piped it but grep found nothing." Errors don't go through pipes by default.

08
Inodes & links
Hard links, soft links, and the inode
Medium
Tasks
→Create a file original.txt, then one hard link and one symlink to it.
→Compare inodes with ls -li — confirm the hard link shares the inode, the symlink does not.
→Delete the original — which link survives, and why?
→Stretch: show what happens when you create a symlink to a file in a different directory, then mv it to a third location.
The mental modelA file is its inode. A name in a directory is just a pointer to one. Hard links add pointers; symlinks are tiny files that hold a path string.

📷
Screenshotcat hard.txt and cat soft.txt after deleting the original.
✍
In reportExplain in your own words why the hard link survives and the symlink breaks. Draw a small ASCII diagram if it helps.

Capstone · multi-tool

Spicy
